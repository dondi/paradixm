Persistent storage is storage that keeps users' files safe. It is also sometimes referred to as non-volatile memory, because unlike volatile memory, or RAM, the storage persists even when the machine is rebooted. The user interface of storage management must be intuitive and efficient so the user can find and manage their files.

Typical Appearance

Persistent storage appears in a variety of forms on devices, and each is targetted towards different needs of users.

The images below are captured in Mac OSX but are typical of many different platforms with minor differences. For more information about platform-specific differences, see "Platform-Specific Instances" below.

File Management

The most common way to navigate through persistent storage is through a file management app, like Finder or File Explorer. It typically has a side bar through which the user can click on folders and navigate to their desired file.

Terminal Window

Another way to navigate through persistent storage is through the terminal window. This is relatively the same across many devices although the commands used to navigate to specific files may differ.

Terminal Window

Another way to manage persistent storage is through a removable drive, such as a USB drive. The user would still manage this through the same finder window or terminal but the different location is worth mentioning.

Typical Behavior

Typical behavior of persistent storage includes the ability to access it in various ways, and its existence on all platforms. When the user saves a file, persistant storage is used. Another example of persistent storage is when a user inserts a usb into their computer and copies or moves files onto their device. The persistant storage is a place for files to be kept until accessed again.

Events

There are a few events that happen to persistent storage in it's various forms.

A few basic events are opening a file, creating a file, deleting a file, and moving a file. Many of these events require an intermediary step that requires the user to do something that technically isn't part of the persistent storage. An example is when the user deletes a file they have to right click the file first and then press the recycle button or something of the sort.

Finding a file is pretty basic the user can usually use a search bar or they can go to the location. They can do this multiple ways such as just looking through finder, in Mac, till they find the the file or doing it through the terminal.

Creating a file is also pretty simple it usually just requires saving a file or downloading a file. Ex. Open a Microsoft file and write the words "Hello World" and then click Save As and save the file where they would like to save it.

Deleting a file is again another easy event where a person usually right clicks the file and then presses "move to trash" or "Delete File" or one can also drag and drop the file into the thrash and it usually is move there.

Copying and pasting a file is another way persistent storage works. One has to first copy the file then navigate through the persistent storage and then paste it where they want into the persistent storage.

Priority Metrics

Persistent storage does have a learnability curve. One must first learn where all the folder and files are located and when doing something such as saving or downloading where the file or folder is placed. Also if the user were to access persistent storage through the terminal using the commands such as 'cd' and 'ls' to find and acquire the file wanted, that would does require a major learning curve. This relates to the memorability metric because the more the user is able to remember where the files are the easier it makes it to find and retrieve. Organization is key here in finding and accessing files quickly.

A big usability metric for persistent storage is errors. The user should be able to navigate through the persistent storage without getting lost and be able to find files easily and quickly. If someone clicks "Downloads" and it takes them to a page that doesn't have the files that were downloaded that would be problematic. A person shouldn't have trouble with errors of finding data in persistent storage if the user interface component is well thoughout and simple.

Organization and guidelines

For the finder make sure your app bundle has the correct filename extension. The Finder looks for the .app filename extension and treats your app appropriately when it finds it. The Finder also shows or hides the filename extension, depending on the state of the "Show all filename extensions" preference in the Advanced pane of Finder preferences.

Use an information property list to supply information to the Finder. The information property list (that is, an Info.plist file) is the standard place to store information about your app and document types. For information on what to put in this file, see Runtime Configuration Guidelines.

Add the appropriate filename extension to documents users can create in your app. Accurate filename extensions help ensure platform interoperability. The user can also set a file type and optionally a creator type for a file, although this is not strictly necessary.

Avoid changing the creator type of existing documents. The creator type implies a distinct sense of ownership over a file. Your app can assign a creator type for files it creates, but it should not change creator types for documents that are created by other apps unless the user gives explicit consent. Note that the user can still associate files with a specific app by using the Info window.

Include a Quick Look generator if your app creates documents in an uncommon or custom format
